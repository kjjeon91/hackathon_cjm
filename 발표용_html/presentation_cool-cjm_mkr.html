해양 RFP 인텔리전스

공고문(RFP)을 “읽고 → 분석하고 → 의사결정”까지 잇는 두 개의 웹 산출물

흩어진 한글파일을 사람이 직접 읽고 정리하던 입찰·R&D 검토 업무를, PDF 자동 파싱 → 구조화 추출 → DB → 시각화 대시보드 → 역할별 정량 의사결정으로 전환했습니다.

팀 · cool-cjm 작성자 · 마경림 (mkr) 담당 · 데이터 파이프라인(agent1·2·DB) + 수주검토 플랫폼 프론트엔드

1 왜 만들게 되었나

기존 위험요소·수주 검토는 “문서 작성 중심”의 수작업이었습니다. 우리는 이걸 “수주 의사결정 중심”의 자동화 체계로 바꾸는 것을 목표로 했습니다.

① 흩어진 문서담당자가 내 PC에서 한글/PDF 공고문을 찾아 열고, 사업개요·예산·기간·과업·평가배점을 직접 읽어 옮겨 적었습니다.
② 주관적 판단참여 여부·위험도 점수가 개인 경험에 의존해 일관성이 떨어지고, 근거 추적이 안 됐습니다.
③ 반복 재작성체크리스트 → 보고서를 매번 다시 쓰고, 결재 자료를 손으로 정리했습니다.

2 두 개의 산출물

하나는 “이미 분석된 RFP를 보여주는” 표출형 대시보드, 다른 하나는 “보면서 참여를 결정하는” 의사결정 플랫폼입니다. 같은 데이터 파이프라인 위에 올라갑니다.

📑

RFP 대시보드

분석된 해양 RFP를 웹으로 보여주는 read-only 대시보드 — 데이터 계층 연동을 시도

  • 3화면 구조 — 문서 목록 → 사업요약+용어해설 → 원본 전문
  • 문서 카드 — 발주처·예산·마감·문서유형(용역/R&D) 배지, 다중선택 필터·검색
  • 사업 요약 — 5줄 요약·핵심 과업·리스크·KPI 덱, 원문 근거 표시
  • AI 어시스턴트 패널 — 검색/LLM은 시각 목업(표출 전용)
시도했지만 채택되진 못함 → 다음엔 이렇게

내가 만든 데이터 계층(agent1 파싱 → agent2 분석 → SQLite DB)과 조회 가이드(query API 9종)를 대시보드에 붙이려 했지만, 최종 결과물 기준에는 못 미쳐 실제로 채택되진 못했습니다.

배운 구조: 다음엔 ① 표현 계층과 데이터 계약(JSON 스키마)을 먼저 합의하고 ② 대시보드가 mock으로 화면을 완성한 뒤 ③ 같은 계약 위에서 mock↔실데이터를 교체하면, 파이프라인 노력이 그대로 화면에 연결됩니다.

🧭

수주검토 의사결정 플랫폼

공고문을 올리면 역할별로 정량 평가하고 자동 결과보고서를 생성하는 React 플랫폼

  • 6단계 플로우 — 업로드+평가자 지정 → 사업개요 → 부서장 → 행정부 → 부문장 → 결과보고서 설정
  • 역할별 정량평가 — 18개 검토항목 + 종합 6축, 레이더 차트 시각화
  • 예산 자동 산정 — 전체 예산 × 부서 예산비율 자동 계산
  • 자동 결과보고서 — 결재란 포함 별도 보고서 + 결재 상신 흐름
내가 한 역할

plus1 명세(.md/.html)를 분석해 designer 서브에이전트에 React 멀티파일 빌드를 위임하고, 이후 cord 스타일 디자인 토큰으로 전 컴포넌트를 리테마(navy/signal-blue) 하며 직접 구현·검증했습니다.

4 시행착오와 해결

막혔던 지점과, 그걸 어떻게 우회/개선했는지가 곧 우리 파이프라인의 설계 근거가 되었습니다.

막힌 지점무엇이 문제였나해결 / 깨달음
로컬 자원 대형 모델 불가 MiMo-V2-Flash(309B)는 GPU 8장 필요 — M1·8GB에서 실행 불가 로컬 소형 모델(Ollama qwen2.5:3b)로 실증 → 완전 무료·오프라인 추론 성공
품질 소형 LLM의 표 추출 3B 모델이 평가배점표를 뭉개고 예산·발주처를 0/빈값으로 채움 HYBRID 전환 — 정형값은 파서+정규식으로 결정적 추출, 서술 요약만 LLM (추출 14→41건)
환경 Python 버전 로컬 Python 3.9는 opendataloader-pdf(3.10+) 미지원 시스템 anaconda python3.11 자동탐지 + run.sh로 환경 고정(재현성↑)
정확도 정규식 오탐 ‘적용 기간(가점기준)’ 표의 “최근 3년 이내”를 사업기간으로 오인식 헤더를 연구개발기간|사업기간|총 기간으로 한정해 오탐 제거
초기 판단 명세 위치 GitHub URL이 404라 명세가 없는 줄 알고 빈 내용으로 덮어쓸 뻔함 로컬에서 실제 18KB 명세 발견 → “파일 미읽음” 가드가 원본 보존
협업 git 충돌 push가 non-fast-forward 거부, 원격이 앞서 있음 원격 겹침 파일이 빈 템플릿임을 검증 후 merge -X ours로 무손실 병합

5 우리의 장점

결정적·재현 가능파서+정규식 기반이라 같은 입력이면 항상 같은 결과. run.sh·requirements·README로 누구나 동일하게 재현합니다.
100% 로컬·무료 동작opendataloader-pdf와 Ollama로 API 키 없이도 풀 파이프라인이 돌아갑니다. 키를 넣으면 코드 수정 없이 Claude LIVE로 전환.
근거 추적 + 할루시네이션 방지전 항목 evidence_id 자동 연결, 근거 없으면 미생성. 배점 합계(100=100) 자동 검산.
계약 기반 확장성같은 DB·JSON 계약 위에 여러 화면을 올릴 수 있는 구조. 의사결정 플랫폼을 완성했고, 표출형 대시보드 연동은 다음 단계의 명확한 길을 확보했습니다.
백엔드 선택 자유AGENT2_BACKEND=hybrid / claude / ollama / offline 한 줄로 교체. 환경·예산에 맞춰 고릅니다.
역할 분리 의사결정부서장·행정부·부문장 판단을 분리해, 결재권자가 리스크 원인을 빠르게 파악합니다.
